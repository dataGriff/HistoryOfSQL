Create Table

Below is the template code for creating a table in SQL.

--This is a comment before creating a table
USE [DatabaseName];
GO

CREATE TABLE [SchemaName].[TableName]
(
[ColumnA] VARCHAR(25) NOT NULL
,[ColumnB] INT NULL
,[ColumnC] DATE NULL
);

The template shows

  • You must state the schema name and table name after your create table statement. Although not mandatory you should always state the schema name otherwise the table name will be created in the default schema of dbo.
  • You must state the name of the first column and its data type.
  • If the column data type is a character field you should state its length.
  • Although not mandatory , you should state whether the column allows NULLs or not, the default is NULL (which allows NULLs).
  • You then repeat these rules for adding extra columns seperated by a comma.
  • You can comment your code by adding two dashes -- and some annotation.

Lets create our first user defined table...

USE [dktest];
GO

--SELECT * FROM [import].[PoundDogs]
--DROP TABLE [import].[PoundDogs]
CREATE TABLE [import].[PoundDogs]
(
--Administrative Columns
[PoundDogsID] INT IDENTITY NOT NULL
,[FirstInserted] DATETIME2 NOT NULL CONSTRAINT DFT_PoundDogs_FirstInserted DEFAULT GETDATE ()
--User Required Columns
,[DogChipNo] INT NOT NULL CONSTRAINT PK_PoundDogs PRIMARY KEY CLUSTERED
,[DogName] VARCHAR(25) NOT NULL
,[DogBreed] VARCHAR(30) NOT NULL
,[DogColour] VARCHAR(20) NOT NULL
,[DogAgeYears] INT NULL
,[DateArrived] DATE NULL
,[TimeArrived] DATE NULL
);

The example adds some handy administrative columns at the start...

  • The ID column with the identity property adds an incrementing integer to the table, handy for seeing the order rows were inserted.
  • The first inserted column with the default constraint ensures that a date and time is added to the row when it is inserted.

The example also shows the addition of a primary key column...

  • The primary key column cannot have any duplicates. This is why chip number is used.
  • The primary key column is how the data is primarily stored, acting as a pointer to each row in the table.
  • Having an integer as a primary key is good practice as it will result in smaller indexes making data retrieval more effective.

There are two handy commented out bits of script at the start...

  • The first is to select all columns from the table so you can check your table once created.
  • The second is to drop the table in case you made a mistake in its creation.
  • This shows that comments are good for having readily available code as well as notes.

The rest of the columns will hold the data we want to store in the table.
